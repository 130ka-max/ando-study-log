# 学習者向けガイド

このガイドは、Codex Study Harnessを使って学習を進める人のための説明です。
細かい内部仕様ではなく、実際に「どう始めて、どう学び、どう終えるか」を中心にまとめています。

Codex Study Harnessは、AIに答えを代わりに出してもらう道具ではありません。
自分が何を考え、どこで迷い、何を確かめ、どう考え直したかを残すための学習記録の仕組みです。

## このハーネスでできること

- 学習テーマをGitHub Issueに残す
- 最初にわからなかったことを書く
- 自分の予想や途中式を残す
- Codexにヒント、解説、添削、類題を出してもらう
- 間違えた原因を整理する
- 最後に確認問題で理解を試す
- 次に復習する内容を決める
- 学習の道すじをGitHub Pagesのレポートにする

大事なのは、正解だけではなく次の流れを残すことです。

```text
問い -> 予想 -> 確認 -> 考え直し -> 気づき -> 次に使う
```

## 最初に準備すること

このリポジトリを自分用にコピーしたら、最初に次を実行します。

```powershell
npm install
npm run check
npm run sync-labels
```

`npm run sync-labels` を使うには、先にGitHub CLIでログインしておきます。

```powershell
gh auth login
```

GitHub Pagesで学習レポートを公開する場合は、GitHubのリポジトリ画面で次を設定します。

```text
Settings -> Pages -> Build and deployment -> Source -> GitHub Actions
```

`Deploy from a branch` ではなく、`GitHub Actions` を選びます。

## 学習を始める

学習を始めるときは、Codexに次のように言います。

```text
ハーネススタート
```

次の言い方でも始められます。

```text
ハーネス開始
学習ハーネスを始めて
このテーマでハーネスを使いたい
```

できれば、開始時に次の情報も一緒に書きます。

- 今日扱う教科やテーマ
- 何ができるようになりたいか
- 今わかっていること
- わからないこと、または間違えた問題
- まず自分で考えたこと
- 希望する学習モード

例:

```text
ハーネススタート
数学の一次関数の文章題を練習したいです。
グラフの読み取りは少しできますが、文章から式を作るところで迷います。
まず「変化の割合」を使えばよいと思いました。
ヒントモードで進めたいです。
```

情報が足りない場合は、Codexが不足分だけ質問します。

## 学習モードを選ぶ

Codexには、進め方を指定できます。

| モード | 使う場面 |
| --- | --- |
| ヒントモード | すぐ答えを見ずに、自分で考えたいとき |
| 添削モード | 自分の解答、途中式、説明を見てもらいたいとき |
| 解説モード | つまずいた内容を順番に説明してほしいとき |
| 類題モード | 似た問題で理解を確認したいとき |
| 振り返りモード | 今日わかったこと、間違いの原因、復習を整理したいとき |
| テストモード | Codexに問題を出してもらい、採点と弱点分析をしたいとき |

迷う場合は、まず `ヒントモード` か `添削モード` を選びます。
問題をまだ解いていないならヒントモード、自分の答えがあるなら添削モードが向いています。

## Issueに何が記録されるか

Issueは、1つの学習テーマを入れる学習カードです。
Codexは、学習開始時に新しいIssueを作るか、既存Issueに追記するかを判断します。

Issueには、主に次の内容を残します。

- 学習テーマ
- 目標
- 最初の理解
- わからなかったこと
- 自分で考えたこと
- Codexとのやりとりで気づいたこと
- 間違えた理由
- 確認問題の結果
- 次に復習すること

開始時には、IssueまたはIssueコメントに必ず次の3つが入ります。

```md
## 問題提起

## 学習者の仮説・考え

## 次の確認問題
```

まだ自分の考えを書いていない場合、Codexは勝手に仮説を作りません。
その場合は「未確認」と記録し、まず自分の予想を書くための確認問題を出します。

## ラベルの意味

Issueラベルは、学習カードにつける目印です。
ハーネス開始時には、Codexがラベルを整理します。

最低限、次のラベルが付きます。

- `type:*`: どんな学習か
- `status:*`: 今どんな状態か
- `portfolio:*`: Webレポートに載せるか

よく使うラベル:

- `type:concept`: 言葉や仕組みを理解する
- `type:practice`: 問題を解いて練習する
- `type:mistake`: 間違えた理由を整理する
- `type:writing`: 作文、レポート、記述を書く
- `status:ready`: 学習を始められる
- `needs:hint`: ヒントがほしい
- `needs:review`: 添削してほしい
- `portfolio:show`: Webレポートに載せる
- `portfolio:hide`: Webレポートに載せない

個人情報、未整理の提出物、公開したくない内容は `portfolio:hide` にします。
通常の学習記録は `portfolio:show` でかまいません。

ラベルの詳しい一覧は [labels.md](labels.md) を確認します。

## 学習中にやること

学習中は、できるだけ次の順で進めます。

1. まず自分の考えを書く
2. Codexからヒント、質問、添削を受ける
3. どこまで正しかったかを確認する
4. どこから誤解していたかを確認する
5. 考えが変わった理由を書く
6. 確認問題または類題を1問以上解く
7. 次に同じ問題を見たときの判断基準を書く

記録としてよい形は、次のような文です。

```text
最初はAと考えたが、Bを確認してCと考えるようになった。
この問題では、AではなくBを分けて考える必要がある。
次に同じ型の問題を見るときは、A、B、Cの順に確認する。
```

短くてもよいので、「最初の見方」「変わった理由」「次に使えること」を残します。

## 学習ログを書く

学習の区切りでは、`learning-log/` にMarkdownの学習ログを残します。
学習ログは、Issueよりもあとから読み返しやすい振り返り用の記録です。

主に次の内容を書きます。

- 関連Issue
- 今日の目標
- 最初の理解
- わからなかったこと
- 自分で考えたこと
- 思考の変化
- 得た知見
- 間違いの原因
- 類題・確認問題の結果
- 次回復習すること
- 次回復習日

雛形は [reflection-template.md](reflection-template.md) を使います。

## 学習を終える

学習を終えるときは、Codexに次のように言います。

```text
ハーネス終了
```

次の言い方でも終了処理に入ります。

```text
ハーネス完了
今日の学習を完了したい
Issueを閉じて次に進みたい
```

ただし、CodexはすぐにIssueを閉じません。
先に、次がそろっているか確認します。

- 学習テーマと目的が記録されている
- 最初にわからなかったことが記録されている
- 自分で考えたことが記録されている
- 思考の変化、得た知見、次に使える判断基準のどれかがある
- 類題または確認問題を1問以上解いている
- 間違いの原因、または誤解しやすい点が記録されている
- 次回復習する内容または日付が決まっている

足りないものがある場合、Issueは閉じません。
Codexは「閉じる前に必要なこと」を短く出します。

## 終了から公開までの流れ

終了条件がそろったら、次の順番で進めます。

```text
1. Issueに終了コメントを残す
2. 必要なら learning-log/ を作成または更新する
3. 思考深化HTMLを生成する
4. 変更をコミットまたはPRにする
5. 学習ブランチをmainへ合流する
6. mainへ合流してからIssueを閉じる
7. Portfolio workflowが動く
8. GitHub Pagesにレポートが公開される
```

順番で特に大事なのは、`main` に合流してからIssueを閉じることです。
Issueを先に閉じると、古い内容でレポートが作られることがあります。

PRを使う場合は、PR本文に次のように書きます。

```text
Closes #<Issue番号>
```

これを書くと、PRが `main` に合流した後でIssueが自動的に閉じます。

## レポートを手動で作る

全体の学習レポートを作る場合:

```powershell
npm run build:portfolio
```

特定のIssueから思考深化レポートを作る場合:

```powershell
npm run harness:end-report -- --issue <Issue番号>
```

学習ログから作る場合:

```powershell
npm run harness:end-report -- --source learning-log/YYYY-MM-DD-topic.md
```

生成物は `public/` に出ます。
`public/` は自動生成される場所なので、通常はコミットしません。

## よく使うコマンド

```powershell
npm install
npm run check
npm run sync-labels
npm run build:portfolio
npm run harness:end-report -- --issue <Issue番号>
```

GitHub Pagesのworkflowを手動で動かす場合:

```powershell
gh workflow run portfolio.yml -f issue_number=<Issue番号>
gh run watch
```

## 困ったとき

`npm run check` が失敗した場合は、表示された不足項目を確認します。
学習ログの見出し不足、確認問題の未実施、ラベルの不足が原因になりやすいです。

Issueを閉じてよいかわからない場合は、[learning-review-checklist.md](learning-review-checklist.md) を確認します。

ラベルの意味がわからない場合は、[labels.md](labels.md) を確認します。

自動記録の詳しい運用を確認したい場合は、[auto-recording-workflow.md](auto-recording-workflow.md) を確認します。

## 大事な約束

このハーネスでは、次のことを大切にします。

- まず自分の考えを書く
- 答えを急がず、ヒントや問いで考える
- 間違いをただ直すだけでなく、原因を説明する
- 確認問題で理解を試す
- 次に復習することを決める
- 自分の言葉で説明できる状態を目指す

学習の記録は、「何をやったか」だけでなく「なぜその考え方の方がよいとわかったか」まで残します。
